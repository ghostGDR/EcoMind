**1. Update __init__() signature:**

```python
def __init__(
    self,
    query_engine: QueryEngine,
    conversation_store: ConversationStore,
    max_history_messages: int = 10  # NEW: Limit history in context
):
    self.max_history_messages = max_history_messages
```

**2. Modify send_message() history formatting:**

Before (Plan 01 implementation):
```python
history = conversation_store.get_conversation(conversation_id)
formatted_history = format_all_messages(history.messages)
```

After (with truncation):
```python
history = conversation_store.get_conversation(conversation_id)
# Truncate to last N messages (keeps recent context only)
recent_messages = history.messages[-self.max_history_messages:] if len(history.messages) > self.max_history_messages else history.messages
formatted_history = format_messages(recent_messages)
```

**3. Context window sizing rationale:**

Default max_history_messages = 10:
- 10 messages = 5 Q&A pairs (5 user + 5 assistant)
- Estimated tokens: ~200 tokens/message * 10 = ~2000 tokens for history
- RAG context from search: ~2500 tokens (5 docs * 500 tokens)
- Total context: ~4500 tokens (well under gpt-4o-mini's 128k limit)
- Leaves room for long user questions and responses

**4. Design decisions:**
- Truncate at message level (not token level) - simpler, preserves complete Q&A pairs
- Keep last N messages (not first N) - recent context more relevant than old context
- Database persistence unchanged - full history always saved, truncation only affects LLM context
- Configurable via constructor - allows tuning per use case (default 10 is conservative)

**5. Edge cases handled:**
- Conversation with 0 messages: No history to truncate, works as before
- Conversation with < max_history_messages: No truncation, includes all history
- Conversation with exactly max_history_messages: No truncation
- Conversation with > max_history_messages: Truncate to last N
