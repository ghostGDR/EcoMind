**Core method**:
```python
def query(self, user_query: str) -> Dict[str, Any]:
    """
    Execute RAG pipeline: retrieve → format context → generate answer
    
    Returns:
    {
        'answer': str,           # LLM-generated response
        'sources': List[Dict],   # Retrieved documents with metadata
        'has_sources': bool      # True if sources found, False if no relevant docs
    }
    """
```

**Implementation steps**:
1. **Retrieve**: Call `search_engine.hybrid_search(user_query, top_k, min_score)`
2. **Check empty**: If no results, return `{'answer': '抱歉，我的知识库中暂时没有关于这个问题的信息。', 'sources': [], 'has_sources': False}`
3. **Format context**: Concatenate retrieved document contents with source labels:
   ```
   [来源 1: {metadata['file_name']}]
   {content}
   
   [来源 2: {metadata['file_name']}]
   {content}
   ```
4. **Build prompt**: Use RAG_PROMPT_TEMPLATE.format(context=formatted_context, query=user_query)
5. **Generate**: Call `llm_client.generate(prompt, temperature=0.7)` with HENRY_SYSTEM_PROMPT as system message
6. **Return**: Package answer with sources for citation extraction in next plan

**Error handling**:
- Catch SearchEngine errors → return error message in 'answer' field
- Catch LLMClient errors → return "生成回答时出错，请稍后重试" message
- Log errors for debugging but don't crash

**Why hybrid_search not semantic_search**: Hybrid search (from Phase 3) combines semantic understanding with keyword boosting for technical terms (ROI, CPC, TikTok), improving accuracy for e-commerce domain queries.

**Why top_k=5, min_score=0.5**: Balance between context richness and relevance. 5 documents provide sufficient context without overwhelming the LLM. 0.5 score threshold filters low-quality matches.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User query → SearchEngine | User input used in vector search queries |
| Retrieved docs → LLM | Knowledge base content sent to external LLM |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | User query manipulation | mitigate | Query passed through SearchEngine's existing validation, no SQL/code injection risk (vector search only) |
| T-04-06 | Information Disclosure | Knowledge base content sent to LLM | accept | User's PROJECT.md explicitly requires cloud LLM API, user controls their own wiki content, no PII in articles |
| T-04-07 | Denial of Service | Large context overwhelming LLM | mitigate | top_k=5 limits retrieved documents, prevents context window overflow (5 docs * ~500 tokens = ~2500 tokens, well under limits) |
| T-04-08 | Spoofing | LLM hallucination (inventing facts) | mitigate | System prompt enforces "只基于提供的知识库内容回答", empty results return explicit "no information" message |
</threat_model>
